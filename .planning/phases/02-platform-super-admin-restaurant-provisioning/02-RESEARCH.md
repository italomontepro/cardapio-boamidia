# Phase 2: Platform Super-Admin — Restaurant Provisioning - Research

**Researched:** 2026-06-15
**Domain:** Next.js 16 Server Actions + Drizzle ORM + Supabase Admin API (service_role), shadcn/ui CRUD admin interface
**Confidence:** HIGH

## Summary

This phase is a CRUD extension of Phase 1's already-established patterns. The schema (`restaurants`, `admin_users`), RLS policies, and the `service_role` admin-client pattern (`scripts/seed.ts`) all already exist and require **no new migration**. The work is almost entirely in `src/app/admin/(dashboard)/`, a new Server Actions module, and `src/lib/auth/actions.ts`.

The one genuinely tricky part is D-07: a single Server Action must create a `restaurants` row (via Drizzle, RLS-permitted because the caller is `super_admin`), an `auth.users` row (via Supabase Admin API `service_role`, **not** part of any Postgres transaction), and an `admin_users` row (via Drizzle or the admin client, bypassing RLS since `admin_users` has no INSERT policy). These three writes span two different systems (Postgres transaction vs. GoTrue HTTP API) and **cannot be wrapped in a single atomic database transaction**. The correct pattern is: wrap the two Postgres writes (`restaurants` + `admin_users`) in `db.transaction()`, but order operations so that if the Auth API call fails, nothing has been written to Postgres yet, and if a later Postgres write fails after the Auth user was created, the orphaned `auth.users` row is cleaned up via `supabaseAdmin.auth.admin.deleteUser()`. This is "atomic-ish" — not a true ACID transaction — and must be documented as such in the plan.

**Primary recommendation:** Create `src/lib/restaurants/actions.ts` with `createRestaurant()` doing: (1) Drizzle transaction inserting `restaurants` row, (2) on success, call `supabaseAdmin.auth.admin.createUser()` with a generated temp password, (3) on Auth success, insert `admin_users` row via Drizzle (same connection, can be inside or right after the transaction since `admin_users` insert needs no RLS bypass at the DB level — Drizzle's runtime client uses a direct Postgres connection, not the RLS-scoped Supabase client, so it's unaffected by RLS either way); (4) on any failure after the Auth user was created, call `supabaseAdmin.auth.admin.deleteUser(userId)` to roll back, and if the `restaurants` row was already committed, delete it too (or run the whole Postgres portion inside one `db.transaction()` that only commits after the Auth call succeeds).

### Pattern 1: service_role Admin Client Factory
**What:** Extract the `supabaseAdmin` client creation from `scripts/seed.ts` into a reusable server-only module so both Server Actions and scripts can use it.
**When to use:** Any server-side code that needs to call `supabase.auth.admin.*` (creating/deleting users) — this bypasses RLS entirely and must NEVER be exposed to client code.
**Example:**
```typescript
// src/lib/supabase/admin.ts
// SERVER-ONLY. Never import from a Client Component.
// Uses SUPABASE_SECRET_KEY (service_role) -- bypasses RLS entirely.
import { createClient } from '@supabase/supabase-js'

export function createAdminClient() {
  return createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.SUPABASE_SECRET_KEY!,
    { auth: { autoRefreshToken: false, persistSession: false } }
  )
}
```
Note: `scripts/seed.ts` also passes `realtime: { transport: ws }` because it runs under plain Node 20 via `tsx` (no native WebSocket). **Server Actions run inside the Next.js server runtime, which has native `fetch`/`WebSocket` support** — the `ws` shim is a `tsx`-script-only workaround and should NOT be carried into `src/lib/supabase/admin.ts`. Verify this during planning by checking Next.js 16's Node target (Node 20+ has `WebSocket` global since v22, but Next.js may polyfill or the realtime transport may simply not be invoked since `auth.admin.*` doesn't use realtime). Recommendation: omit the `realtime` option in the app's admin client — it's only needed because `auth.admin.createUser` internally constructs a full client; if a runtime error about `WebSocket` appears, add the same `ws` transport as a fallback.

### Pattern 2: Atomic-ish Restaurant + Admin Creation (D-07/D-08/D-09)
**What:** A single Server Action performs 3 writes across 2 systems (Postgres via Drizzle, Auth via Supabase Admin API), with manual compensation on partial failure.
**When to use:** `createRestaurant()` Server Action, the core of PLAT-05.

**Recommended order of operations** (chosen to minimize orphaned state and make cleanup simple):

```typescript
// src/lib/restaurants/actions.ts
'use server'

import { db } from '@/db'
import { restaurants, adminUsers } from '@/db/schema'
import { createAdminClient } from '@/lib/supabase/admin'
import { eq } from 'drizzle-orm'

export async function createRestaurant(input: CreateRestaurantInput) {
  // 1. Validate input (zod) -- name, slug, adminEmail
  const parsed = createRestaurantSchema.safeParse(input)
  if (!parsed.success) return { error: parsed.error.flatten() }
  const { name, slug, adminEmail } = parsed.data

  // 2. Pre-check slug uniqueness for a fast, friendly error (D-05).
  //    NOTE: this is a TOCTOU race -- the DB unique constraint on
  //    restaurants.slug is the real guarantee. Catch the unique
  //    violation below as the authoritative check.
  const existing = await db.query.restaurants.findFirst({
    where: eq(restaurants.slug, slug),
  })
  if (existing) {
    return { error: { slug: ['Este link já está em uso. Escolha outro slug para o restaurante.'] } }
  }

  // 3. Insert restaurant row FIRST (Postgres only -- cheap to roll back
  //    if the Auth API call in step 4 fails).
  let restaurant
  try {
    [restaurant] = await db.insert(restaurants).values({ name, slug }).returning()
  } catch (err) {
    // Unique violation race (D-05 authoritative check) -- Postgres error code 23505
    if (isUniqueViolation(err)) {
      return { error: { slug: ['Este link já está em uso. Escolha outro slug para o restaurante.'] } }
    }
    throw err
  }

  // 4. Create the auth.users row via service_role (D-08/D-09).
  //    This is OUTSIDE any Postgres transaction -- if it fails, we must
  //    manually delete the restaurant row created in step 3.
  const tempPassword = generateTempPassword()
  const supabaseAdmin = createAdminClient()
  const { data: authData, error: authError } = await supabaseAdmin.auth.admin.createUser({
    email: adminEmail,
    password: tempPassword,
    email_confirm: true, // D-09
  })

  if (authError || !authData.user) {
    // Rollback step 3.
    await db.delete(restaurants).where(eq(restaurants.id, restaurant.id))
    return { error: { adminEmail: ['Não foi possível criar o usuário administrador. Verifique o e-mail.'] } }
  }

  // 5. Insert admin_users row (Postgres only, Drizzle runtime client --
  //    NOT RLS-scoped, so no policy issue inserting role=restaurant_admin).
  try {
    await db.insert(adminUsers).values({
      userId: authData.user.id,
      role: 'restaurant_admin',
      restaurantId: restaurant.id,
    })
  } catch (err) {
    // Rollback BOTH: delete the auth user AND the restaurant row.
    await supabaseAdmin.auth.admin.deleteUser(authData.user.id)
    await db.delete(restaurants).where(eq(restaurants.id, restaurant.id))
    throw err
  }

  // 6. Success -- return temp password ONCE (D-08). Caller is
  //    responsible for displaying it and NOT persisting it.
  return {
    success: true,
    restaurant: { id: restaurant.id, name: restaurant.name, slug: restaurant.slug },
    tempPassword,
    adminEmail,
  }
}
```

**Why this order:** Step 3 (restaurants insert) is the cheapest to undo (single Postgres DELETE, no external API call). Step 4 (Auth API) is the step most likely to fail for user-input reasons (duplicate email, invalid email format) — failing here only requires undoing step 3. Step 5 (admin_users insert) is extremely unlikely to fail (both FKs just succeeded), but if it does, both prior steps must be undone.

**Alternative considered — wrap steps 3+5 in `db.transaction()`:** This would mean step 3 doesn't commit until step 5 succeeds, so a failure in step 5 wouldn't need an explicit `db.delete()` for the restaurant row (transaction auto-rollback). However, step 4 (the Auth API call) would then need to happen **between** the transaction's insert statements — which is fine, `db.transaction()` just holds the connection open; the HTTP call to GoTrue can happen inside the callback. **Recommendation for the planner: use `db.transaction()` wrapping steps 3 and 5, with step 4 (Auth API call) inside the transaction callback between them.** This reduces the manual-rollback surface to just "delete the orphaned auth user if the transaction itself throws after step 4 succeeded" — one compensation path instead of two.

```typescript
// Refined: db.transaction() wraps the two Postgres writes
const tempPassword = generateTempPassword()
const supabaseAdmin = createAdminClient()
let createdAuthUserId: string | null = null

try {
  const result = await db.transaction(async (tx) => {
    const [restaurant] = await tx.insert(restaurants).values({ name, slug }).returning()

    const { data: authData, error: authError } = await supabaseAdmin.auth.admin.createUser({
      email: adminEmail, password: tempPassword, email_confirm: true,
    })
    if (authError || !authData.user) {
      throw new ProvisioningError('admin_create_failed', authError?.message)
    }
    createdAuthUserId = authData.user.id

    await tx.insert(adminUsers).values({
      userId: authData.user.id, role: 'restaurant_admin', restaurantId: restaurant.id,
    })

    return { restaurant, authUser: authData.user }
  })

  return { success: true, restaurant: result.restaurant, tempPassword, adminEmail }
} catch (err) {
  // If the auth user was created but the transaction threw AFTER that
  // (e.g., admin_users insert failed), the Postgres writes are rolled
  // back automatically, but the auth.users row is NOT -- clean it up.
  if (createdAuthUserId) {
    await supabaseAdmin.auth.admin.deleteUser(createdAuthUserId)
  }
  if (err instanceof ProvisioningError && err.code === 'admin_create_failed') {
    return { error: { adminEmail: ['Não foi possível criar o usuário administrador. Verifique o e-mail.'] } }
  }
  if (isUniqueViolation(err)) {
    return { error: { slug: ['Este link já está em uso. Escolha outro slug para o restaurante.'] } }
  }
  throw err
}
```

This refined version has exactly **one** compensation path (delete orphaned auth user) instead of two (delete auth user + delete restaurant), because the Postgres transaction handles the restaurants/admin_users rollback automatically. **This is the recommended pattern for the plan.**

### Pattern 3: `is_active` Login Gate (D-11)
**What:** After `login()` resolves `role`/`restaurant_id` from `admin_users`, additionally check `restaurants.is_active` for `restaurant_admin` users.
**When to use:** Modify `src/lib/auth/actions.ts` `login()`.
**Example:**
```typescript
// src/lib/auth/actions.ts -- login(), after the existing adminRow lookup
if (!adminRow) {
  await supabase.auth.signOut()
  redirect('/admin/login?error=not_an_admin')
}

// D-11: restaurant_admin login gate -- super_admin has no restaurant_id, unaffected.
if (adminRow.role === 'restaurant_admin') {
  const { data: restaurant } = await supabase
    .from('restaurants')
    .select('is_active')
    .eq('id', adminRow.restaurant_id)
    .single()

  if (!restaurant?.is_active) {
    await supabase.auth.signOut()
    redirect('/admin/login?error=restaurant_inactive')
  }
}

redirect(adminRow.role === 'super_admin' ? '/admin' : '/painel')
```
Note: the existing `login()` SELECT on `admin_users` only fetches `role` (line 23 of `src/lib/auth/actions.ts`). It must be extended to also select `restaurant_id` to support this check — currently only `getCurrentAdmin()` in `session.ts` selects `restaurant_id`. Both queries run through the RLS-scoped `@supabase/ssr` client; the existing "admins read own row" SELECT policy on `admin_users` covers this, and the `restaurants` SELECT policy (`is_super_admin() or id = current_admin_restaurant_id()`) covers the `restaurants.is_active` read for a `restaurant_admin` reading their own restaurant.

**Login page error message:** `/admin/login?error=restaurant_inactive` needs a corresponding PT-BR message added to `src/app/admin/login/page.tsx`'s error-rendering logic (not in UI-SPEC's copy table since UI-SPEC focuses on the admin dashboard, not the login page — flag this as a small addition the planner should account for). Suggested text: "Este restaurante está desativado. Contate o administrador da plataforma."

### Pattern 4: Admin Count Column (D-01, Claude's Discretion)
**What:** The `restaurants` table needs a "Admins" column showing `count(admin_users where restaurant_id = restaurants.id)`.
**When to use:** PLAT-04 table listing.
**Example (Drizzle relational query with count via SQL):**
```typescript
// src/app/admin/(dashboard)/page.tsx -- using Drizzle's runtime client
import { db } from '@/db'
import { restaurants, adminUsers } from '@/db/schema'
import { sql, eq } from 'drizzle-orm'

const rows = await db
  .select({
    id: restaurants.id,
    name: restaurants.name,
    slug: restaurants.slug,
    isActive: restaurants.isActive,
    createdAt: restaurants.createdAt,
    adminCount: sql<number>`count(${adminUsers.id})`.as('admin_count'),
  })
  .from(restaurants)
  .leftJoin(adminUsers, eq(adminUsers.restaurantId, restaurants.id))
  .groupBy(restaurants.id)
  .orderBy(restaurants.name)
```
**Important caveat:** Using `db` (Drizzle runtime client, direct Postgres connection via `DATABASE_URL_RUNTIME`) **bypasses RLS entirely** — it connects as the Postgres role configured in the connection string, not as the authenticated `auth.uid()`. The existing `page.tsx` (Phase 1) deliberately uses the **`@supabase/ssr`-backed `createClient()`** (the anon-key, RLS-scoped client) specifically so RLS enforces "super_admin sees all restaurants" (see comment at `src/app/admin/(dashboard)/page.tsx` lines 7-9 — "This query IS the D-09 proof that super_admin sees all restaurants").

**Recommendation:** Keep using the RLS-scoped `@supabase/ssr` client for the main `restaurants` SELECT (preserves the Phase 1 security proof), and do the admin-count either as (a) a second RLS-scoped query against `admin_users` grouped client-side in JS, or (b) a Postgres view/RPC if a join is preferred. Given `admin_users` SELECT policy is "admins read own row **or** super_admin reads all" — `is_super_admin()` covers reading ALL `admin_users` rows too, so a second `supabase.from('admin_users').select('restaurant_id')` query (no `.eq` filter) works under RLS and can be grouped/counted in JS. This avoids introducing a Drizzle/RLS-bypass query into a page that Phase 1 deliberately kept RLS-scoped for its security proof. **Do not use the Drizzle `db` client for this page's primary listing query** — reserve `db`/Drizzle for the Server Action writes (Pattern 2), where RLS bypass via `service_role` is already required for `admin_users` writes anyway, OR note that `restaurants` writes (insert/update) ARE covered by the `for all` RLS policy for `is_super_admin()` — meaning the Drizzle runtime client's writes to `restaurants` will succeed regardless of RLS (Drizzle connects with elevated Postgres credentials, RLS is bypassed for that role) — but `admin_users` writes still need to happen through `service_role`-equivalent privilege, which the Drizzle runtime connection already has (it's a privileged Postgres role, separate from the `anon`/`authenticated` RLS roles). **Net effect: Drizzle writes in Server Actions work for both tables regardless of RLS** (this is why Phase 1 used Drizzle for schema/seed writes) — RLS only matters for the `@supabase/ssr`-backed reads used in pages/`login()`.

### Anti-Patterns to Avoid
- **Treating D-07 as a true ACID transaction:** It is impossible — `auth.users` lives in Supabase's managed Auth schema/GoTrue service, reachable only via HTTP API, not via the app's Postgres connection. Document "atomic-ish with compensation" explicitly in the plan; don't claim full atomicity.
- **Persisting the temp password:** D-08 explicitly forbids storing it in plain text anywhere (no DB column, no log line, no session storage beyond the single render). Returning it directly in the Server Action's response object (held only in React state for the success dialog) is correct; writing it to `admin_users` or any audit table is not.
- **Using the Drizzle runtime client for the `restaurants` listing page's primary query:** breaks the Phase 1 RLS security proof pattern (see Pattern 4).
- **Client-side slug uniqueness as the only check:** D-05's zod validation in the form is a UX nicety; the DB unique constraint on `restaurants.slug` (already present per Phase 1 schema) is the authoritative guard against race conditions — the Server Action must catch the Postgres unique-violation error too (Pattern 2, step 3's catch block).
- **Re-confirming "Ativar":** Per UI-SPEC copywriting contract, only "Desativar" requires `AlertDialog` confirmation (D-12). "Ativar" is a direct action — don't add a confirmation dialog to it.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Slug generation from PT-BR restaurant names | Custom regex/normalize chain | `slugify` npm package | Correctly handles NFD normalization + diacritic stripping (ã→a, ç→c, etc.) + lowercasing + dash-collapsing in one call — a hand-rolled version needs the same logic anyway |

## Common Pitfalls

### Pitfall 1: Treating `db.transaction()` + Auth API call as fully atomic
**What goes wrong:** Planner assumes "wrap everything in `db.transaction()`" means the `auth.users` row is also rolled back if the transaction throws.
**Why it happens:** `db.transaction()` only controls the Postgres connection's BEGIN/COMMIT/ROLLBACK. The `supabaseAdmin.auth.admin.createUser()` call is a separate HTTP request to GoTrue — once it returns success, that row exists in `auth.users` regardless of what happens to the Postgres transaction afterward.
**How to avoid:** Use Pattern 2's refined approach — call the Auth API *inside* the `db.transaction()` callback (between the two Postgres inserts), track the created `userId`, and in the `catch` block (outside the transaction), explicitly call `supabaseAdmin.auth.admin.deleteUser(userId)` if it was created but the transaction ultimately failed.
**Warning signs:** Integration test creates a restaurant with a duplicate-but-different-slug scenario that fails on the `admin_users` insert (e.g., a contrived FK violation) — check Supabase Auth dashboard afterward for an orphaned user with no matching `admin_users` row.

### Pitfall 2: `prepare: false` + transaction pooler — prepared statement errors on `db.transaction()`
**What goes wrong:** Drizzle's `db.transaction()` could theoretically issue prepared statements internally, which fail against Supabase's transaction-mode pooler (port 6543).
**Why it happens:** PgBouncer transaction mode historically didn't support prepared statements at all; `prepare: false` (already set in `src/db/index.ts`) disables this in postgres-js.
**How to avoid:** No action needed — `src/db/index.ts` already sets `prepare: false` on the `postgres()` client used by `db`, and `db.transaction()` respects this. Confirmed via Drizzle's official Supabase connection docs (HIGH confidence — `prepare: false` is the documented fix). This is a non-issue for this phase given the existing setup.
**Warning signs:** Errors like `prepared statement "..." already exists` or `ERROR: prepared statement does not exist` — would indicate `prepare: false` was somehow not applied to the connection used inside `db.transaction()`. Should not occur given current `src/db/index.ts`.

### Pitfall 3: `admin_users` INSERT via Drizzle runtime client — is RLS actually bypassed?
**What goes wrong:** Planner worries the `admin_users` insert in Pattern 2 will fail because `admin_users` has only a SELECT RLS policy (per `0002_rls_policies.sql` and CONTEXT.md canonical refs).
**Why it happens:** Confusion between the two Supabase access paths: (1) `@supabase/ssr` client (anon/authenticated Postgres role, RLS enforced), vs (2) Drizzle's `db` client (`DATABASE_URL_RUNTIME`, connects as the `postgres` superuser-equivalent role via the pooler, RLS does NOT apply to this role by default in Postgres — RLS only restricts non-superuser roles unless `FORCE ROW LEVEL SECURITY` is set, which Phase 1's migrations did not set).
**How to avoid:** Verify empirically during planning/execution (not just by reading docs) — `scripts/seed.ts` already proves this works: it does `db.insert(adminUsers).values(...)` directly (line 39-43, 50-52, 59-61) with no RLS errors, against the same `DATABASE_URL_RUNTIME` connection. Pattern 2 follows the exact same proven path. **HIGH confidence** — this is empirically verified by existing working code in this repo, not just theory.
**Warning signs:** If an `admin_users` insert via `db` ever throws a permission/RLS error, the Postgres connection string's role has changed (unlikely) — would be a Phase-1-environment regression, not a Phase 2 design flaw.

### Pitfall 4: Slug uniqueness check is a TOCTOU race under concurrent creation
**What goes wrong:** Two super admins (or double-submit) create restaurants with the same slug at nearly the same time; both pass the pre-check SELECT, both attempt INSERT, one fails on the unique constraint.
**Why it happens:** The pre-check (Pattern 2 step 2) and the INSERT (step 3) are not atomic relative to each other.
**How to avoid:** The pre-check is a UX nicety (fast feedback before hitting the Auth API). The INSERT's unique constraint on `restaurants.slug` (already present from Phase 1 schema: `slug: text('slug').notNull().unique()`) is authoritative — catch the Postgres `23505` error in the `catch` block and return the same D-05 error message. Pattern 2 already includes both checks.
**Warning signs:** None expected in normal single-super-admin usage (low concurrency for this admin tool), but the code path must exist for correctness.

### Pitfall 5: `email_confirm: true` and Supabase project email settings
**What goes wrong:** Some Supabase projects have "Confirm email" toggled in Auth settings such that `email_confirm: true` in `createUser()` is the override needed to skip the confirmation email — if this setting interacts unexpectedly, the admin might receive a confirmation email anyway, or login might still be blocked pending confirmation.
**Why it happens:** `email_confirm: true` in the Admin API `createUser()` call explicitly marks the user's email as confirmed at creation time, which should make `auth.users.email_confirmed_at` non-null immediately — this is the documented behavior and is exactly what `scripts/seed.ts` already relies on (Phase 1's seeded admins log in without email confirmation, per `verify-auth.ts` passing).
**How to avoid:** No new action — D-09 explicitly says to replicate `scripts/seed.ts`'s exact pattern, which is already verified working (AUTH-02 passed in Phase 1 via `verify-auth.ts`). **HIGH confidence** — empirically proven in this codebase.
**Warning signs:** If a newly-provisioned restaurant admin reports "check your email to confirm" on login attempt, compare the `createUser()` call against `scripts/seed.ts` for a missed `email_confirm: true`.

### Pitfall 6: `login()` query needs `restaurant_id`, currently only selects `role`
**What goes wrong:** D-11's `is_active` check needs `adminRow.restaurant_id`, but the current `login()` query (`src/lib/auth/actions.ts` line 20-24) only does `.select('role')`.
**Why it happens:** Simple oversight risk when adding the new check without updating the existing SELECT.
**How to avoid:** Change `.select('role')` to `.select('role, restaurant_id')` in `login()`. Flagged explicitly in Pattern 3 above.
**Warning signs:** TypeScript error / runtime `undefined` when accessing `adminRow.restaurant_id` if the SELECT isn't updated.

### slugify usage for D-04

```typescript
// src/lib/restaurants/slug.ts
import slugify from 'slugify'

export function generateSlug(name: string): string {
  return slugify(name, { lower: true, strict: true, locale: 'pt' })
}
// generateSlug('Pizzaria do João') -> 'pizzaria-do-joao'
// `strict: true` strips characters not matching [A-Za-z0-9-], collapsing
// repeated dashes -- matches D-04's "Gerado automaticamente a partir do
// nome" requirement with no extra config needed.
```

### Temp password generation for D-08
```typescript
// src/lib/restaurants/actions.ts (helper)
import { randomBytes } from 'crypto'

function generateTempPassword(): string {
  // 16 hex chars (8 bytes) -- meets typical Supabase min-length (6) with
  // margin; alphanumeric only avoids URL/copy-paste issues in the
  // success dialog. Adjust length/charset if Supabase project password
  // policy requires symbols.
  return randomBytes(8).toString('hex')
}
```

### Detecting unique constraint violation with postgres-js
```typescript
// src/lib/restaurants/actions.ts (helper)
function isUniqueViolation(err: unknown): boolean {
  // postgres-js (the driver behind drizzle-orm/postgres-js) attaches the
  // Postgres error code to thrown errors as `.code`. 23505 = unique_violation.
  return (
    typeof err === 'object' &&
    err !== null &&
    'code' in err &&
    (err as { code: string }).code === '23505'
  )
}
```

## Open Questions

1. **Does `createAdminClient()` need the `ws`/`realtime` transport shim from `scripts/seed.ts` when run inside a Next.js Server Action?**
   - What we know: `scripts/seed.ts` needs `ws` because it runs under plain Node 20 via `tsx`, and supabase-js's client constructor requires a `WebSocket` global even when realtime is unused.
   - What's unclear: Whether the Next.js 16 server runtime (Node 20+, possibly with polyfills) already provides a global `WebSocket`, making the `ws` import unnecessary in `src/lib/supabase/admin.ts`.
   - Recommendation: Try without the `ws` transport first (cleaner). If `createAdminClient()` throws a `WebSocket is not defined` error at runtime, add the same `ws` shim as `scripts/seed.ts`. This is a 5-minute empirical check during the first task that exercises this code path — low risk, easy to detect and fix.

2. **Supabase project password policy minimum requirements for `auth.admin.createUser()`**
   - What we know: `scripts/seed.ts` uses `'changeme123!'` (11 chars, mixed case + digit + symbol) successfully.
   - What's unclear: The exact minimum length/complexity enforced by this Supabase project's Auth settings — a 16-hex-char password (Code Examples) is alphanumeric-only with no uppercase/symbol, which might not satisfy a stricter policy if one exists.
   - Recommendation: Either (a) check the Supabase dashboard Auth settings for password policy during planning, or (b) generate a temp password that satisfies a reasonably strict policy by construction (e.g., mix of uppercase, lowercase, digit, symbol) regardless — safer default. Suggest: `` `${randomBytes(6).toString('hex')}${'Aa1!'}` `` style construction, or use a small helper that guarantees at least one of each character class.
