# Phase 04.1: Localização de unidades via mapa - lat/lng e formulário em etapas no admin - Context

**Gathered:** 2026-06-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Esta fase (inserida, urgente) entrega a capacidade do admin do restaurante marcar a localização geográfica exata (latitude/longitude) de cada unidade, através de um formulário de cadastro/edição de unidade redesenhado em etapas, com uma etapa dedicada de seleção no mapa. Substitui o Dialog único de campo simples da Phase 3 por um wizard de 3 etapas. Não inclui: lógica de sugestão de unidade mais próxima no cardápio público (isso é Phase 5, D-03 — consome o lat/lng que esta fase produz, mas não a implementa).

**Goal:** O admin do restaurante consegue definir a localização precisa de cada unidade através de um formulário em etapas com seleção interativa no mapa, fornecendo o dado de lat/lng que a Fase 5 depende para sugerir a unidade mais próxima ao cliente.

</domain>

<decisions>
## Implementation Decisions

### Provedor de mapa
- **D-01:** Leaflet + OpenStreetMap como base do mapa — gratuito, sem API key, sem custo recorrente. Geocoding de busca de endereço via Nominatim (OSM), também gratuito.

### Estrutura das etapas do formulário
- **D-02:** Formulário em 3 etapas dentro do mesmo Dialog:
  1. **Básico** — nome da unidade.
  2. **Contato/Horário** — WhatsApp, endereço (texto livre, campo já existente), horários de funcionamento.
  3. **Localização** — mapa interativo para marcar a posição exata.
- **D-03:** Navegação via botões "Próximo"/"Voltar" no rodapé do Dialog, com indicador de progresso (steps) mostrando em qual etapa o admin está (ex: "Etapa 2 de 3").

### Interação com o mapa
- **D-04:** Campo de busca de endereço na etapa de Localização, usando geocoding via Nominatim, que centraliza o mapa no resultado; depois disso, um pin arrastável permite o admin ajustar a posição exata manualmente.
- **D-05:** Ao entrar na etapa de Localização, o sistema já geocodifica automaticamente o endereço digitado na etapa 2 (Contato/Horário) e centraliza o mapa nesse ponto — o admin só precisa ajustar o pin se a posição não estiver exata, ou buscar de novo se quiser outro ponto.

### Unidades sem localização (retrofit e novas)
- **D-06:** Unidades criadas antes desta fase (Phase 3) ficam com `lat`/`lng` null. A tabela de unidades (`/painel/unidades`) mostra um badge discreto "Sem localização" nessas linhas, incentivando o admin a editar e completar — nada é bloqueado.
- **D-07:** Para unidades novas, a etapa de Localização é **opcional** — o admin pode avançar e salvar sem marcar um ponto no mapa. A unidade fica com `lat`/`lng` null e recebe o mesmo badge "Sem localização" de D-06. Consistente com D-06, não trava o cadastro rápido.
- **D-08 (consequência para Fase 5):** Unidades com `lat`/`lng` null são simplesmente ignoradas pela lógica de sugestão de proximidade da Fase 5 (D-03) — sem erro, sem fallback especial.

### Claude's Discretion
- Biblioteca React para Leaflet (`react-leaflet` ou wrapper customizado sobre Leaflet vanilla).
- Texto/estilo exato do indicador de progresso das etapas (bolinhas, barra, texto "Etapa N de 3").
- Tratamento de erro de geocoding (endereço não encontrado pelo Nominatim) — fallback para centro padrão (ex: centro do Brasil) + aviso ao admin.
- Estilo visual exato do badge "Sem localização" na tabela de unidades.
- Migration: nomes exatos das colunas (`lat`/`lng` ou `latitude`/`longitude`), tipo Postgres (`numeric` ou `double precision`).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `react-hook-form` + `zodResolver` já é o padrão de formulário do projeto (Phase 2/3) — o wizard de 3 etapas deve reutilizar esse padrão, navegando entre subconjuntos de campos do mesmo form ao invés de 3 forms separados.
- `libphonenumber-js` já tratado no formulário atual para o campo WhatsApp — não muda nesta fase.
- shadcn `Dialog` já instalado e usado para o form atual — o wizard continua dentro do mesmo `Dialog`, só adicionando navegação interna por etapas.

### Established Patterns
- Server Actions com `'use server'` + validação zod + Drizzle insert/update + `revalidatePath('/painel/unidades')` — `createUnit`/`updateUnit` seguem esse padrão e devem continuar seguindo ao adicionar lat/lng.
- RLS já escopa as queries por `restaurantId` automaticamente — nenhuma mudança de RLS necessária para esta fase (lat/lng são só novas colunas na tabela já protegida).

### Integration Points
- Nenhuma lib de mapa (Leaflet, Mapbox, Google Maps) está instalada — precisa adicionar `leaflet` + `react-leaflet` (ou equivalente) como nova dependência.
- Nenhuma lib de geocoding está instalada — Nominatim é uma API HTTP pública (sem SDK necessário), chamada via `fetch` direto.

</code_context>

<specifics>
## Specific Ideas

- Usuário pediu explicitamente "estilo iFood" para a experiência de localização (contexto vindo da Fase 5) — aqui no admin, isso se traduz em busca de endereço + pin ajustável, dando ao admin uma forma rápida e visual de fixar a posição da unidade.
- Formulário deve ficar "bem bonito" — wizard de 3 etapas com indicador de progresso visual é a tradução dessa pedida em decisão concreta (D-02/D-03).

</specifics>
