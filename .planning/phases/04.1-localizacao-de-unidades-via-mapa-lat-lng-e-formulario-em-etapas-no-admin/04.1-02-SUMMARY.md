# Phase 04.1 Plan 02: Server-Layer Lat/Lng Wiring + Nominatim Geocoding Summary

**Extended upsertUnitSchema with optional/nullable lat/lng, wired createUnit/updateUnit to persist coordinates, and shipped a server-only Nominatim geocoding helper plus a geocodeUnitAddress Server Action for the upcoming map wizard.**

## Performance

- **Duration:** 4 min
- **Started:** 2026-06-17T06:16:53Z
- **Completed:** 2026-06-17T06:20:14Z
- **Tasks:** 2 completed
- **Files modified:** 3 (1 created, 2 modified)

## Accomplishments
- `upsertUnitSchema` now accepts optional/nullable `lat` (-90..90) and `lng` (-180..180), keeping Step 3 of the unit wizard skippable (D-07) while rejecting out-of-range coordinates
- `createUnit` and `updateUnit` both persist `lat`/`lng` from validated input, defaulting to `null` when omitted — existing Phase-3 units remain valid
- New `src/lib/units/geocode.ts`: server-only `geocodeAddress()` wrapper around Nominatim's `/search` endpoint with the mandatory custom `User-Agent` header (Nominatim rejects default UAs), mapping the API's `lon` field to our `lng`
- New `geocodeUnitAddress` Server Action in `actions.ts`: auth-guarded (`getCurrentAdmin()`), trims and validates the address, returns `{success, lat, lng, displayName}` for a resolved address or a friendly `{error}` otherwise
- `scripts/verify-units-location.ts` now fully green including the previously-skipped live `GEOCODE-SUCCESS`/`GEOCODE-NOTFOUND` assertions against the real Nominatim API
