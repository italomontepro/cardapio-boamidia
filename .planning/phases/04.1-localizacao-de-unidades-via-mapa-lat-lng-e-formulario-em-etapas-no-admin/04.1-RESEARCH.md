# Phase 04.1: Localização de unidades via mapa - Research

**Researched:** 2026-06-17
**Domain:** Leaflet/React-Leaflet map integration in Next.js 16 App Router, Nominatim geocoding, Drizzle migration for geo columns, multi-step react-hook-form wizard
**Confidence:** HIGH

## Summary

This phase adds an optional lat/lng pair to the `units` table and turns the existing single-step unit Dialog into a 3-step wizard, with the final step embedding a Leaflet map for pin placement, backed by Nominatim for address-based geocoding. All four sub-problems (map rendering in App Router, geocoding policy compliance, column typing, and wizard UX inside one `useForm`) are well-trodden and have a single clearly-dominant idiomatic solution — no exploratory tradeoffs needed, the locked decisions (D-01 through D-08) map directly onto standard patterns.

The two technical risks that actually matter for planning are: (1) Leaflet's reliance on `window`/`document` requires the map to be loaded as a client-only dynamic import (`next/dynamic` with `ssr: false`) — a plain `'use client'` directive on the component is NOT sufficient by itself if any ancestor renders during SSR with the Leaflet instance constructed at module scope; and (2) Nominatim's public endpoint enforces a strict 1 req/sec limit and mandates a real identifying `User-Agent`/`Referer` — this must be called from a Next.js Server Action (or Route Handler), never directly from the browser, both to satisfy the policy correctly and to avoid the admin's browser UA being silently rejected.

**Primary recommendation:** Use `leaflet` 1.9.4 + `react-leaflet` 5.0.0 (React 19-compatible), load the map view via `next/dynamic(() => import(...), { ssr: false })` from a `'use client'` wrapper, geocode through a new Server Action that calls Nominatim server-side with a descriptive `User-Agent` header, store `lat`/`lng` as nullable `numeric(10, 7)` columns with Drizzle's `{ mode: 'number' }`, and build the wizard as step-scoped field subsets of the existing single `upsertUnitSchema` validated via `form.trigger([...fieldNames])` on "Next".

### Pattern 1: Client-only Leaflet map via next/dynamic
**What:** Leaflet constructs DOM nodes and reads `window`/`document` at import time (e.g., `L.Icon.Default` references `window.location` for icon URL resolution in some builds). In Next.js App Router, Server Components render on the server first; if the Leaflet module executes during that SSR pass — even inside a `'use client'` component — it can throw `window is not defined` or `ReferenceError: document is not defined` during the server-render of the client component's first pass (Next.js does still execute client components' render function on the server during the initial HTML generation, even though their effects don't run there).

**When to use:** Always, for any component that imports `react-leaflet` or `leaflet` directly.

**Example:**
```typescript
// unit-location-map-loader.tsx — Server Component or shared file, no 'use client' needed here
'use client'
import dynamic from 'next/dynamic'

export const UnitLocationMap = dynamic(
  () => import('./unit-location-map').then((m) => m.UnitLocationMap),
  {
    ssr: false,
    loading: () => (
      <div className="flex h-64 items-center justify-center rounded-md border bg-muted text-sm text-muted-foreground">
        Carregando mapa…
      </div>
    ),
  }
)
```
```typescript
// unit-location-map.tsx — the real Leaflet component, also 'use client'
'use client'
import { MapContainer, TileLayer, Marker, useMapEvents } from 'react-leaflet'
import L from 'leaflet'
import 'leaflet/dist/leaflet.css'
import iconUrl from 'leaflet/dist/images/marker-icon.png'
import iconRetinaUrl from 'leaflet/dist/images/marker-icon-2x.png'
import shadowUrl from 'leaflet/dist/images/marker-shadow.png'

// Fix default marker icon paths broken by bundlers (webpack AND Turbopack) — see Pitfall 1
delete (L.Icon.Default.prototype as { _getIconUrl?: unknown })._getIconUrl
L.Icon.Default.mergeOptions({ iconUrl, iconRetinaUrl, shadowUrl })

export function UnitLocationMap({
  lat, lng, onChange,
}: { lat: number; lng: number; onChange: (lat: number, lng: number) => void }) {
  function DraggableMarker() {
    useMapEvents({}) // map ref available if needed for recentering
    return (
      <Marker
        position={[lat, lng]}
        draggable
        eventHandlers={{
          dragend: (e) => {
            const { lat: newLat, lng: newLng } = e.target.getLatLng()
            onChange(newLat, newLng)
          },
        }}
      />
    )
  }

  return (
    <MapContainer center={[lat, lng]} zoom={15} style={{ height: 280, width: '100%' }}>
      <TileLayer
        attribution='&copy; OpenStreetMap contributors'
        url="https://{s}.tile.openstreetmap.org/{z}/{x}/{y}.png"
      />
      <DraggableMarker />
    </MapContainer>
  )
}
```
Source: pattern synthesized from official react-leaflet docs (`MapContainer`/`Marker`/`useMapEvents` API) + the well-documented Leaflet+bundler marker-icon workaround (GitHub `PaulLeCam/react-leaflet#808`).

### Pattern 2: Server-side Nominatim geocoding via Server Action
**What:** Geocoding must happen server-side, both because Nominatim's usage policy requires a real identifying `User-Agent`/`Referer` (browsers cannot set custom `User-Agent` on `fetch`, and stock UAs are explicitly rejected) and because doing it server-side gives a single, easily-rate-limited call path.

**When to use:** Step 3 entry (D-05 auto-geocode) and the manual address-search field (D-04).

**Example:**
```typescript
// src/lib/units/geocode.ts — server-only module
const NOMINATIM_URL = 'https://nominatim.openstreetmap.org/search'

export interface GeocodeResult { lat: number; lng: number; displayName: string }

export async function geocodeAddress(query: string): Promise<GeocodeResult | null> {
  const url = `${NOMINATIM_URL}?format=jsonv2&limit=1&q=${encodeURIComponent(query)}`
  const res = await fetch(url, {
    headers: {
      // Required by Nominatim usage policy — stock fetch UA is rejected.
      'User-Agent': 'CardapioBoaMidia/1.0 (contato@suporte.folkscomp.com)',
    },
  })
  if (!res.ok) return null
  const data = (await res.json()) as Array<{ lat: string; lon: string; display_name: string }>
  if (data.length === 0) return null
  return { lat: parseFloat(data[0].lat), lng: parseFloat(data[0].lon), displayName: data[0].display_name }
}
```
```typescript
// src/lib/units/actions.ts — exposed as a Server Action callable from the client step component
'use server'
import { geocodeAddress } from './geocode'

export async function geocodeUnitAddress(address: string) {
  if (!address.trim()) return { error: 'Endereço vazio.' }
  const result = await geocodeAddress(address)
  if (!result) return { error: 'Endereço não encontrado.' }
  return { success: true, ...result }
}
```
Response shape and required headers confirmed against official Nominatim docs (`nominatim.org/release-docs/latest/api/Search/`) and the OSM Foundation's Nominatim Usage Policy (`operations.osmfoundation.org/policies/nominatim/`).

### Pattern 3: Multi-step wizard with one useForm + step-scoped trigger()
**What:** Keep a single `useForm<UpsertUnitInput>()` instance for the whole Dialog (matches existing codebase convention of one `zodResolver(upsertUnitSchema)` per form). Track current step in local `useState<number>`. On "Próximo", call `await form.trigger([...fieldsForCurrentStep])` — this validates only the named fields against the full schema and returns `false` if any fail, blocking advancement without needing separate per-step schemas or `.pick()`.

**When to use:** This phase's exact case — 3 steps, one Dialog, one submit at the end.

**Example:**
```typescript
const STEP_FIELDS = {
  1: ['name'] as const,
  2: ['whatsappNumber', 'address', 'hours'] as const,
  3: ['lat', 'lng'] as const, // optional fields, trigger() still safe to call (no-op if no rules fail)
}

const [step, setStep] = useState(1)
const form = useForm<UpsertUnitInput>({ resolver: zodResolver(upsertUnitSchema), defaultValues: {...} })

async function goNext() {
  const valid = await form.trigger(STEP_FIELDS[step as 1 | 2 | 3])
  if (!valid) return
  setStep((s) => Math.min(s + 1, 3))
}
function goBack() { setStep((s) => Math.max(s - 1, 1)) }
```
Only the final step's "Salvar"/"Criar Unidade" button has `type="submit"`; all "Próximo"/"Voltar" buttons are `type="button"` to avoid premature form submission (this mirrors the existing single-step Dialog's `<form onSubmit={handleSubmit(onSubmit)}>` wrapper — the `<form>` tag stays around all 3 steps, only its visible content changes per step).

### Anti-Patterns to Avoid
- **Separate `useForm()` per step / separate Zod schemas per step merged at the end:** Adds state-merging complexity (combining 3 independent form states into one payload) for no benefit here — the existing single-schema pattern (`upsertUnitSchema`) already covers all fields; just extend it with optional `lat`/`lng` and gate advancement with `trigger()`.
- **Importing `leaflet` or `react-leaflet` directly in a Server Component or anywhere reachable during the SSR pass without `next/dynamic({ ssr: false })`:** Triggers `window`/`document is not defined` even with `'use client'` on the importing file, because Next.js still renders client components on the server for the initial HTML.
- **Calling Nominatim directly from client-side `fetch`:** Browsers cannot override the `User-Agent` header (it's a forbidden header name per the Fetch spec), so the stock browser UA gets sent — explicitly rejected by Nominatim's policy ("stock User-Agents as set by http libraries will not do"). Always proxy through a Server Action/Route Handler.
- **Re-geocoding on every keystroke in the address-search field:** Violates the 1 req/sec policy almost immediately. Debounce (e.g., 500-800ms) or require explicit "Buscar" button click (D-04 describes a search field, not live-as-you-type — a button-triggered search is simplest and policy-safe).

## Common Pitfalls

### Pitfall 1: Broken/invisible default marker icons under Turbopack/webpack
**What goes wrong:** The pin marker renders as a broken image icon (or nothing at all) because Leaflet's CSS (`leaflet.css`) references marker PNGs via relative paths that bundlers don't resolve correctly, and `L.Icon.Default`'s internal `_getIconUrl()` method assumes a non-bundled file layout.
**Why it happens:** Leaflet was originally designed for a plain `<script>`/CDN setup where its CSS and images sit side-by-side on a static path. Module bundlers (webpack, and Turbopack inherits the same class of issue) process `leaflet.css`'s `url(...)` references and the JS's dynamic icon URL construction differently, breaking the implicit path assumption.
**How to avoid:** In the client-only map component (the one dynamically imported with `ssr: false`), explicitly delete `L.Icon.Default.prototype._getIconUrl` and call `L.Icon.Default.mergeOptions({ iconUrl, iconRetinaUrl, shadowUrl })` with the three marker images imported via ES module `import` statements (not `require`), so the bundler emits proper asset URLs. See Pattern 1 code example above.
**Warning signs:** Map tiles load fine but the marker pin shows as a small broken-image icon, or no marker is visible at all; browser devtools Network tab shows 404s for `marker-icon.png` at an unexpected path.

### Pitfall 2: SSR crash from importing Leaflet without `ssr: false`
**What goes wrong:** Build or runtime error `window is not defined` / `document is not defined`, sometimes only surfacing in production build (`next build`) and not in `next dev`, or vice versa.
**Why it happens:** Next.js App Router still executes the render function of `'use client'` components on the server during the initial HTML pass (hydration requires matching server-rendered HTML). If `leaflet`'s module-scope code (or `react-leaflet`'s `MapContainer`) touches `window`/`document` synchronously during that render, it throws server-side even though the component is marked `'use client'`.
**How to avoid:** Wrap the import in `next/dynamic(() => import('./unit-location-map'), { ssr: false })`, and only render that dynamic component from inside Step 3 — never import `react-leaflet` types/components directly into any file that's part of the SSR render tree without going through the dynamic wrapper.
**Warning signs:** Error message explicitly naming `window`/`document`/`navigator` as undefined, often pointing into `node_modules/leaflet/dist/leaflet-src.js`.

### Pitfall 3: Nominatim rate-limit / policy violation
**What goes wrong:** Requests start returning HTTP 403/429 or get silently blocked ("Usage limit reached") after a burst of geocoding calls (e.g., admin clicking "Buscar" repeatedly, or auto-geocoding firing on every keystroke).
**Why it happens:** The public `nominatim.openstreetmap.org` endpoint enforces an absolute max of 1 request/second per application, and requires a real identifying `User-Agent` or `Referer` — generic/stock UAs are rejected outright, and repeated identical queries without caching can get a client blocked as "faulty."
**How to avoid:** (1) Call Nominatim only from the server (Server Action), never client-side — this also solves the custom-`User-Agent` requirement, since browsers can't set arbitrary `User-Agent` headers anyway. (2) Trigger geocoding only on explicit user actions (entering Step 3 once per session per D-05, or an explicit "Buscar" button click per D-04) — never on every keystroke; add a debounce if any live-search UI is added later. (3) Set a real, descriptive `User-Agent` string identifying the app (e.g., `"CardapioBoaMidia/1.0 (contact-email-or-url)"`).
**Warning signs:** Geocoding silently stops working for all units after testing it repeatedly in a short window during development.

### Pitfall 4: Drizzle `numeric` columns returning strings, breaking lat/lng math
**What goes wrong:** Without specifying a mode, Drizzle's `numeric()` column type maps to TypeScript `string` (Postgres `numeric` is arbitrary-precision and JS `number` can't safely represent all values), so `unit.lat` ends up as `"−23.5505"` (a string) instead of a number — breaking any arithmetic (e.g., Haversine distance in Phase 5) and requiring manual `parseFloat` everywhere it's read.
**Why it happens:** This is Drizzle's deliberate, safe default for `numeric`/`decimal` to avoid precision loss — but GPS coordinates never need arbitrary precision, so this default is unnecessarily inconvenient here.
**How to avoid:** Use `numeric('lat', { precision: 10, scale: 7, mode: 'number' })` — the `mode: 'number'` option (available since Drizzle ORM 0.41.0, well within this project's 0.45.2) makes Drizzle parse the column as a JS `number` at the type and runtime level, while Postgres still stores it as exact fixed-point `numeric`.
**Warning signs:** TypeScript infers `lat: string | null` instead of `lat: number | null` on the `units` select type; runtime string concatenation bugs when lat/lng are used in template literals or comparisons.

### Drizzle migration: nullable lat/lng columns
```typescript
// src/db/schema.ts — add to the existing `units` pgTable definition
export const units = pgTable('units', {
  id: uuid('id').primaryKey().defaultRandom(),
  restaurantId: uuid('restaurant_id').notNull().references(() => restaurants.id, { onDelete: 'cascade' }),

  address: text('address'),

  // far more than needed but consistent with the project's existing numeric('price', {...}) convention.
  lat: numeric('lat', { precision: 10, scale: 7, mode: 'number' }),
  lng: numeric('lng', { precision: 10, scale: 7, mode: 'number' }),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),

  unique('units_restaurant_slug_unique').on(table.restaurantId, table.slug),
  index('idx_units_restaurant_id').on(table.restaurantId),

```
Run `drizzle-kit generate` (against `DATABASE_URL`, Session Mode/5432 per project convention) to produce `0003_*.sql` — this will be a simple additive `ALTER TABLE units ADD COLUMN lat numeric(10,7); ADD COLUMN lng numeric(10,7);`, no RLS or data migration needed since both are nullable with no default.

**Precision rationale:** `numeric(10, 7)` allows up to 3 integer digits (longitude can be up to ±180) and 7 decimal digits — 7 decimal places of lat/lng precision is ~1.1cm at the equator, more than sufficient for "which building" precision; this matches common GPS coordinate storage conventions and is more than enough headroom versus the simpler `double precision` alternative the user could also pick (both are valid; `numeric` chosen here for consistency with the existing `price` column's use of `numeric` in this codebase, not a precision-driven decision since `double precision` would also work fine for ~15-17 significant decimal digits).

## Open Questions

   - What we know: CONTEXT.md explicitly defers this naming choice to discretion.
   - What's unclear: No existing convention in `schema.ts` to anchor on (no prior geo columns).
   - Recommendation: Use short `lat`/`lng` (matches common API/library conventions like Leaflet's own `LatLng` object having `.lat`/`.lng` properties — using the same names reduces mental mapping overhead when passing data directly to `<Marker position={[unit.lat, unit.lng]}>`).
