Revisiting this value would mean reworking Phase 5 UI, which is not in scope here.

## Interaction & State Notes (executor reference)

These supplement the visual contract with the specific states the executor must implement, derived directly from CONTEXT.md decisions:

1. **Send button visibility logic** (D-02, D-09, D-11): visible only when `cart.items.length > 0 AND unit.whatsappNumber !== null`. Three mutually exclusive footer states: (a) empty cart тЖТ no footer CTA at all, just the existing empty-state message; (b) cart has items + no `whatsappNumber` тЖТ red warning text replaces the button; (c) cart has items + `whatsappNumber` present тЖТ full footer (name, delivery tabs, send button, clear-cart button).
2. **Send action**: builds the `wa.me` URL via a new pure helper (`src/lib/menu/whatsapp.ts` per CONTEXT.md Integration Points), `encodeURIComponent`-encodes the message, opens it (Claude's Discretion: use `window.open(url, '_blank')` to avoid navigating away from the menu page on mobile тАФ preserves cart state visibly in the background tab), then immediately fires the `sonner` success toast. Cart is NOT cleared (D-06).
3. **Message format** (D-05): plain text, no emojis. Structure: title line (unit/restaurant name) тЖТ optional name line тЖТ optional delivery-type line тЖТ itemized list (`{qty}x {name} - {price}`, with notes indented on the following line) тЖТ subtotal line at the end. Reuse `formatBRL` from `src/lib/menu/format.ts` for every price shown in the message, exactly as already used in `cart-sheet.tsx`.
4. **Clear-cart action**: dispatches a new `CLEAR` action on the existing `cart-provider.tsx` reducer (additive тАФ `ADD`/`SET_QTY`/`REMOVE`/`HYDRATE` stay unchanged), gated behind the `AlertDialog` confirm step.
5. **Large-cart / accents / emoji rendering** (D-10): no truncation logic in the UI layer тАФ `wa.me` is trusted to handle long messages; this is validated empirically on a real device per Roadmap success criterion 4, not via a UI state.
