# Phase 06 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **Scope note:** Phase 6 does NOT introduce a new screen. It extends the existing `CartSheet` component (`src/app/r/[restaurantSlug]/[unitSlug]/cart-sheet.tsx`) built in Phase 5 with a footer section (name field, pickup/delivery selector, send button, clear-cart button) plus a toast confirmation and an "unidade sem WhatsApp" inline warning. All tokens below govern the **new Phase 6 footer elements** specifically; where Phase 5 tokens are reused as-is they are marked accordingly.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding (e.g. `gap-1`) |
| sm | 8px | Compact element spacing (e.g. `gap-2`); **Phase 6 NEW:** spacing between a footer field's label and its input/control (name label→input, delivery label→tabs) |
| md | 16px | Default element spacing (`px-4` sheet padding); **Phase 6 NEW:** spacing between distinct footer sections (name field block → delivery-selector block → send button → clear-cart button) |
| lg | 24px | Section padding (footer block separation from the cart item list, via existing `Separator`) |
| xl | 32px | Layout gaps (not used inside the Sheet — too tall for a bottom sheet) |
| 2xl | 48px | Major section breaks (not used in this phase) |
| 3xl | 64px | Page-level spacing (not used in this phase) |

**Phase 6 new footer block — exact rule:** the new footer container uses `flex flex-col gap-4` (16px) as the outer stack between the name field, delivery-selector, send button, and clear-cart button. Inside each label+control pair (e.g. `<Label>` + `<Input>`, `<Label>` + `<Tabs>`), use `flex flex-col gap-2` (8px) — tighter, since a label sits directly above its own control. Both values are on the standard 8pt scale {4, 8, 16, 24, 32, 48, 64}. This replaces the previous `gap-3` (12px) draft, which was off-scale.

Exceptions:
- Icon-only buttons (`icon-sm` stepper +/− in `CartSheet`) use the existing `size-7` (28px) shadcn token — below the 44px touch-target guideline, but this is the **established Phase 5 pattern**, not a new exception. Do not change it in this phase (out of scope — would require revisiting Phase 5 UI).
- Phase 5's pre-existing cart item list in `cart-sheet.tsx` uses `gap-3` (12px) between cart lines. That is a pre-existing Phase 5 fact, untouched and out of scope for Phase 6 — Phase 6 does not modify it and does not inherit it into the new footer block (see rule above: the new footer block uses `gap-2`/`gap-4` only).

## Typography

**Phase 6 new footer elements — exactly 2 weights:**

| Role | Size | Weight | Line Height | Applies to (Phase 6 new elements) |
|------|------|--------|-------------|------------------------------------|
| Body / regular | 14px (`text-sm`) | 400 (regular) | 1.5 | Name input label, name input text, delivery-selector label, `TabsTrigger` text, clear-cart button label, toast body text, unit-has-no-WhatsApp warning text |
| Emphasis | 16px (`text-base`) for subtotal / 18px (`text-lg`, shadcn `SheetTitle` default) for the sheet title | 600 (`font-semibold`) | 1.5 (subtotal) / 1.2 (title) | Send button label ("Enviar pedido via WhatsApp"), subtotal row, Sheet title (unchanged, pre-existing) |

No Phase 6 new element uses `font-medium` (500). Where a label needs visual distinction from its value (e.g. "Seu nome (opcional)" vs. the typed value), use the 400-weight `Label` component's default muted-foreground color for hierarchy instead of a heavier weight — color and size carry the distinction, not weight.

**Pre-existing Phase 5 fact (outside this contract, not governed by it):** `cart-sheet.tsx` item names elsewhere in the same Sheet (the cart item list, built in Phase 5 and not touched by Phase 6) use `font-medium` (500) as an established convention. That pre-existing usage is unchanged by this phase and is not part of the Phase 6 typography contract above.

## Interaction & State Notes (executor reference)

These supplement the visual contract with the specific states the executor must implement, derived directly from CONTEXT.md decisions:

1. **Send button visibility logic** (D-02, D-09, D-11): visible only when `cart.items.length > 0 AND unit.whatsappNumber !== null`. Three mutually exclusive footer states: (a) empty cart → no footer CTA at all, just the existing empty-state message; (b) cart has items + no `whatsappNumber` → red warning text replaces the button; (c) cart has items + `whatsappNumber` present → full footer (name, delivery tabs, send button, clear-cart button).
2. **Send action**: builds the `wa.me` URL via a new pure helper (`src/lib/menu/whatsapp.ts` per CONTEXT.md Integration Points), `encodeURIComponent`-encodes the message, opens it (Claude's Discretion: use `window.open(url, '_blank')` to avoid navigating away from the menu page on mobile — preserves cart state visibly in the background tab), then immediately fires the `sonner` success toast. Cart is NOT cleared (D-06).
3. **Message format** (D-05): plain text, no emojis. Structure: title line (unit/restaurant name) → optional name line → optional delivery-type line → itemized list (`{qty}x {name} - {price}`, with notes indented on the following line) → subtotal line at the end. Reuse `formatBRL` from `src/lib/menu/format.ts` for every price shown in the message, exactly as already used in `cart-sheet.tsx`.
4. **Clear-cart action**: dispatches a new `CLEAR` action on the existing `cart-provider.tsx` reducer (additive — `ADD`/`SET_QTY`/`REMOVE`/`HYDRATE` stay unchanged), gated behind the `AlertDialog` confirm step.
5. **Large-cart / accents / emoji rendering** (D-10): no truncation logic in the UI layer — `wa.me` is trusted to handle long messages; this is validated empirically on a real device per Roadmap success criterion 4, not via a UI state.
